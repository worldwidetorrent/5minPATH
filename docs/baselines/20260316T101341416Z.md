## Replay regime comparison

The first three-regime comparison for this session is:

- [`run_20260317T022654Z`](/home/ubuntu/testingproject/artifacts/replay_compare/2026-03-16/session_20260316T101341416Z/run_20260317T022654Z)

It was run with `snapshot_cadence_ms: 1000` to match the capture granularity. For this dataset, the default 250 ms replay cadence oversamples stale 1-second capture state and adds a large compute penalty without adding new information.

Command:

```bash
.venv/bin/python -m rtds.cli.compare_replay_regimes \
  --date 2026-03-16 \
  --session-id 20260316T101341416Z \
  --config configs/replay/comparison_1s.yaml \
  --rebuild-reference true \
  --rebuild-snapshots true
```

Comparison verdict:

- good-only and good+degraded are not economically interchangeable
- all-windows is identical to good+degraded because this session has no `unusable` windows
- degraded windows should remain excluded or be subdivided further before policy extraction

Top-line regime results:

- `good_only`: `3600` snapshots, `2867` trades, `hit_rate = 0.5629`, `average_selected_net_edge = 0.1786`, `total_pnl = 101.583`, `average_roi = 0.5637`
- `good_plus_degraded`: `21900` snapshots, `18012` trades, `hit_rate = 0.6811`, `average_selected_net_edge = 0.1244`, `total_pnl = 963.597`, `average_roi = 0.3236`
- `all_windows`: identical to `good_plus_degraded`

Interpretation:

- degraded windows do not merely add harmless noise; they materially change trade mix and edge profile
- average selected edge and average ROI compress sharply once degraded windows are included
- because `all_windows == good_plus_degraded`, the remaining issue is degraded-window quality, not a hidden unusable-window tail in this session
