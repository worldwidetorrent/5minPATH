## 1. Purpose

This document defines the canonical schema spine for the `testingproject` codebase. It is the shared grammar used by all downstream modules: ingestion, normalization, mapping, snapshots, replay, evaluation, and execution.

The goal is not to describe every table in the system. The goal is to define the small set of shared entities, identifiers, timestamp semantics, units, naming rules, and invariants that every other schema must obey.

If a later schema contradicts this document, this document wins unless an explicit architecture decision record supersedes it.

## 2. Scope

This document covers:

- canonical entities
- canonical identifiers
- canonical timestamp semantics
- canonical price and size semantics
- units and storage conventions
- naming rules
- versioning rules
- cross-table relationship rules

This document does not define:

- venue-specific raw payloads
- full normalized event tables
- window reference rules
- replay feature columns
- execution simulation logic

Those are covered in downstream schema documents.

## 3. Design principles

### 3.1 Identity is separate from observation

An entity identifier names a thing. An observation describes that thing at a point in time. For example:

- `window_id` names a 5-minute interval
- `market_id` names a Polymarket market
- `snapshot_id` names a derived observation of state

These identifiers must never be reused to mean one another.

### 3.2 Event time is separate from receive time

The system must preserve at least source event time and local receive time for any event-like record. This project depends on timestamp alignment, so time semantics are first-class.

### 3.3 Price semantics must be explicit

The project may store many distinct kinds of price. Generic columns named `price` are forbidden in normalized, reference, feature, and snapshot schemas unless the table is truly single-purpose and the meaning is encoded in the table name.

### 3.4 UTC everywhere

All persisted timestamps are UTC. Any display-layer timezone conversion is downstream and never part of stored canonical state.

### 3.5 Derived logic is versioned

Any table or file generated by code must carry version tags for the logic that produced it.

## 4. Canonical entities

These entities are shared across the project.

### 4.1 Asset

A tradable or referenced underlying, such as BTC.

Required canonical attributes:

- `asset_id`
- `asset_class`
- `base_symbol`
- `quote_symbol` when applicable
- `display_name`
- `status`

Initial supported values:

- `BTC`

### 4.2 Venue

A data source or execution venue.

Examples:

- Binance
- Coinbase
- Kraken
- OKX
- Bybit
- Polymarket
- Chainlink

Required canonical attributes:

- `venue_id`
- `venue_type`
- `display_name`
- `status`

### 4.3 Instrument

A venue-scoped instrument or feed.

Examples:

- Binance BTCUSDT spot
- Coinbase BTC-USD spot
- Chainlink BTC/USD stream
- a specific Polymarket binary contract

Required canonical attributes:

- `instrument_id`
- `venue_id`
- `asset_id`
- `instrument_type`
- `venue_symbol`
- `quote_ccy`
- `status`

### 4.4 Market

A tradable market at the execution venue. In phase 1, this mainly refers to a specific Polymarket contract.

Required canonical attributes:

- `market_id`
- `venue_id`
- `asset_id`
- `market_type`
- `title`
- `status`

### 4.5 Oracle feed

A settlement-relevant price source.

Required canonical attributes:

- `oracle_feed_id`
- `venue_id`
- `asset_id`
- `feed_name`
- `feed_type`
- `status`

### 4.6 Window

A canonical resolution interval. In phase 1, one exact 5-minute BTC interval.

Required canonical attributes:

- `window_id`
- `asset_id`
- `window_type`
- `window_start_ts`
- `window_end_ts`
- `duration_seconds`

### 4.7 Snapshot

A derived row representing the knowable state of a specific market/window at a specific time.

Required canonical attributes:

- `snapshot_id`
- `window_id`
- `market_id`
- `snapshot_ts`
- `schema_version`
- `feature_version`

### 4.8 Order

A submitted or simulated trading instruction.

Required canonical attributes:

- `order_id`
- `market_id`
- `window_id` when known
- `side`
- `order_type`
- `created_ts`
- `status`

### 4.9 Fill

A completed or simulated execution against an order.

Required canonical attributes:

- `fill_id`
- `order_id`
- `market_id`
- `fill_ts`
- `fill_price`
- `fill_size`
- `fee_amount`

## 5. Canonical identifiers

## 5.1 Rules

All canonical identifiers must be:

- deterministic or traceable
- unique within their entity namespace
- stable over time
- string-serializable
- safe for parquet partitioning, filenames, and joins

Identifiers must not encode mutable attributes beyond what is explicitly defined.

## 5.2 Required identifier formats

### `asset_id`

Type: string  
Pattern: uppercase short code  
Examples:

- `BTC`
- `ETH`

### `venue_id`

Type: string  
Pattern: lowercase snake or plain lowercase  
Examples:

- `binance`
- `coinbase`
- `kraken`
- `okx`
- `bybit`
- `polymarket`
- `chainlink`

### `instrument_id`

Type: string  
Pattern: `<venue_id>:<instrument_type>:<venue_symbol>`

Examples:

- `binance:spot:BTCUSDT`
- `coinbase:spot:BTC-USD`
- `chainlink:stream:BTC-USD`
- `polymarket:binary:<market_slug_or_id>`

Note: `instrument_id` is not necessarily the execution market. A `market_id` is still used for venue-native market identity.

### `market_id`

Type: string  
Pattern: venue-native identifier stored canonically as string

Examples:

- Polymarket condition ID or market ID rendered as string

If the venue provides multiple IDs, store the canonical internal `market_id` plus additional native IDs in side columns.

### `window_id`

Type: string  
Pattern: `<asset_id_lower>-5m-<YYYYMMDDTHHMMSSZ>`

Example:

- `btc-5m-20260313T120500Z`

The timestamp component is the canonical UTC window start.

### `oracle_event_id`

Type: string nullable  
Pattern: venue-native round/sequence/event identifier if available

If no source identifier exists, do not fabricate one. Use null and rely on time plus lineage fields.

### `snapshot_id`

Type: string  
Pattern: deterministic composite key or content hash of `(window_id, market_id, snapshot_ts, snapshot_build_version)`

Recommended human-readable form for phase 1:

- `snap:<window_id>:<market_id>:<YYYYMMDDTHHMMSSmmmZ>`

### `order_id`

Type: string  
Pattern: system-generated UUID or deterministic simulation ID

### `fill_id`

Type: string  
Pattern: system-generated UUID or deterministic simulation ID

## 6. Timestamp semantics

## 6.1 Allowed timestamp columns

The following names are canonical. Use them exactly when applicable.

### `event_ts`

Definition: timestamp asserted by the source for when the event occurred.  
Type: UTC timestamp with millisecond or finer precision if available.

### `recv_ts`

Definition: local UTC timestamp when the collector process received the message.  
Type: UTC timestamp.

### `proc_ts`

Definition: local UTC timestamp when the event was normalized or persisted.  
Type: UTC timestamp.

### `snapshot_ts`

Definition: the as-of UTC timestamp for a snapshot row.  
Type: UTC timestamp.

### `window_start_ts`

Definition: canonical UTC start of the resolution window.

### `window_end_ts`

Definition: canonical UTC end of the resolution window.

### `created_ts`, `updated_ts`, `fill_ts`, `cancel_ts`

Definitions: lifecycle timestamps for execution records.

## 6.2 Time precision

Persist timestamps at microsecond precision if the stack supports it cleanly. Millisecond precision is acceptable if consistent end-to-end. Do not mix timestamp precision arbitrarily across related tables.

## 6.3 Duration and age columns

Use explicit units in the column name.

Allowed examples:

- `seconds_remaining`
- `chainlink_current_age_ms`
- `composite_max_age_ms`
- `gap_duration_ms`

Forbidden examples:

- `age`
- `latency`
- `duration`

## 6.4 Clock requirements

Collectors should have access to:

- wall-clock UTC timestamps for persistence
- monotonic local timing for measuring internal elapsed durations

Monotonic clocks must never be persisted as absolute timestamps.

## 7. Canonical price semantics

## 7.1 Price column rules

Every price column must encode its meaning in the name.

Preferred names:

- `trade_price`
- `best_bid`
- `best_ask`
- `mid_price`
- `microprice`
- `chainlink_open_anchor_price`
- `chainlink_current_price`
- `chainlink_settle_price`
- `composite_now_price`
- `up_bid`
- `up_ask`
- `down_bid`
- `down_ask`
- `fill_price`

Forbidden names in non-raw tables:

- `price`
- `px`
- `mark`

unless the table is scoped so tightly that ambiguity is impossible and the choice is documented.

## 7.2 Price units

### Spot/oracle prices

Unit: decimal USD per BTC  
Storage: numeric decimal preferred; float acceptable only in transient compute layers, not in long-lived canonical datasets unless explicitly documented.

### Binary market prices

Unit: decimal probability-like price in `[0, 1]`  
Examples:

- 0.57 means 57 cents
- 0.99 means 99 cents

Do not store binary prices as integer cents in canonical schemas.

## 7.3 Derived price conventions

### `mid_price`

Definition:

`(best_bid + best_ask) / 2`

### `spread`

For spot/oracle contexts, use either:

- `spread_abs`
- `spread_bps`

For binary markets, use either:

- `spread_abs`
- `spread_cents`

Do not overload one spread column across incompatible units.

## 8. Size and quantity semantics

## 8.1 Column naming

Use explicit size semantics.

Examples:

- `trade_size_base`
- `trade_notional_quote`
- `bid_size_base`
- `ask_size_base`
- `up_bid_size_contracts`
- `up_ask_size_contracts`

## 8.2 Unit rules

### Spot venues

Base-size columns use asset units, e.g. BTC.  
Quote notional columns use USD or stablecoin quote as applicable.

### Binary markets

Contract size columns must specify whether they represent:

- shares/contracts
- USD notional equivalent

For phase 1, prefer explicit contract/share count plus optional notional estimate.

## 9. Status and enum conventions

Use closed enums wherever possible. Free-text statuses are discouraged.

Recommended shared enums include:

- `status`: `active`, `inactive`, `delisted`, `unknown`
- `venue_type`: `spot_exchange`, `prediction_market`, `oracle`, `synthetic`
- `instrument_type`: `spot`, `binary`, `stream`, `index`, `metadata`
- `market_type`: `binary_updown_5m`
- `window_type`: `updown_5m`
- `side`: `buy`, `sell`
- `order_type`: `market`, `limit`, `ioc`, `fok`, `simulation`
- `mapping_status`: `mapped`, `ambiguous`, `missing_anchor`, `missing_settle`, `inactive`

## 10. Nullability principles

Use null only when a value is genuinely unknown, unavailable, or inapplicable.

Do not encode missingness with magic numbers such as:

- `0`
- `-1`
- empty string

If nulls are common for an important field, define a companion status field.

Example:

- `chainlink_open_anchor_price` nullable
- `chainlink_open_anchor_status` non-null enum

## 11. Schema versioning

The following version columns are canonical in derived tables:

- `schema_version`
- `normalizer_version`
- `feature_version`
- `simulation_version`

Rules:

- bump `schema_version` when columns or semantics change
- bump `normalizer_version` when raw-to-normalized transformation changes
- bump `feature_version` when derived feature logic changes
- bump `simulation_version` when fill/fee/slippage assumptions change

## 12. Relationship rules

These are the minimum required joins.

- an `instrument` belongs to one `venue` and one `asset`
- a `market` belongs to one `venue` and one `asset`
- a `window` belongs to one `asset`
- a `snapshot` references one `window` and one `market`
- an `order` references one `market`; it may reference one `window`
- a `fill` references one `order` and one `market`

A `market` may map to one canonical `window` in the phase-1 use case.

## 13. Partitioning and file layout rules

Canonical datasets should be partitionable by UTC date.

Recommended partition columns:

- `date_utc` derived from the relevant as-of timestamp
- optional `venue_id` or `table_type` where helpful

Do not partition on high-cardinality IDs such as `snapshot_id` or `order_id`.

## 14. Naming rules

### 14.1 General format

Use `snake_case` for:

- column names
- schema fields
- python attributes
- parquet columns

### 14.2 Suffix conventions

- `_id` for identifiers
- `_ts` for timestamps
- `_ms` for millisecond durations/ages
- `_seconds` only for constants or metadata, not time-left fields; prefer `seconds_remaining`
- `_price` for spot/oracle prices
- `_bid` / `_ask` for market quotes
- `_size_*` for sizes
- `_flag` for booleans
- `_status` for enums
- `_version` for lineage versions

### 14.3 Boolean naming

Use positive boolean names.

Examples:

- `chainlink_gap_flag`
- `composite_stale_flag`
- `market_tradable_flag`

Avoid double negatives.

## 15. Data lineage requirements

Any derived table must be reproducible from source tables plus configuration and code version.

Minimum lineage fields for derived outputs:

- `schema_version`
- `source_build_id` or equivalent run identifier
- relevant logic version fields
- source date range or snapshot date

## 17. Invariants

The following invariants must hold unless a status field explicitly marks an exception:

1. `window_end_ts > window_start_ts`
2. `duration_seconds = window_end_ts - window_start_ts`
3. all persisted timestamps are UTC
4. all binary quote fields are in `[0, 1]`
5. `best_ask >= best_bid` when both are non-null
6. `up_ask >= up_bid` when both are non-null
7. `down_ask >= down_bid` when both are non-null
8. `seconds_remaining >= 0`
9. any row with a null anchor or settle value must carry a non-null status explaining why

## 18. Open questions

These are intentionally left to downstream ADRs or schema documents:

- exact storage type choice for decimals across parquet writers
- whether `snapshot_id` should remain human-readable or move to hashed form
- whether phase 1 needs a separate `oracle_round_id` abstraction beyond source-native event IDs
- whether `microprice` belongs in the canonical feature vocabulary or only in execution-focused schemas

## 19. Summary

This document defines the shared vocabulary of the system. All downstream schema documents must build on these rules:

- explicit identity
- explicit time semantics
- explicit price semantics
- UTC-only storage
- versioned derived logic
- deterministic joins through canonical IDs

The next schema documents should define:

1. how markets map to windows and anchors
2. how replay snapshots materialize knowable state at each moment
